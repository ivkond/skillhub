# Phase 6 Plan 01: Theme Core And Policy Guard

<objective>
Build the token foundation and enforcement mechanisms for the new color system so all downstream UI migration work is constrained to semantic tokens.
</objective>

<wave_boundary>
Wave 1 deliberately excludes full visual migration of `button`, `card`, `input`, and `tabs`.
This wave delivers foundation contracts only: semantic tokens, bootstrap/theme helpers, policy gate semantics, and tests proving primitive contract readiness.
Primitive visual migration is deferred to Plan 02 to keep Wave 1 execution risk bounded and auditable.
</wave_boundary>

    Add `web/src/shared/theme/theme-preference.ts` with concrete API:
    - `getStoredThemePreference()` reading `localStorage['skillhub-theme']` and returning `light|dark|system` (default `system`).
    - `resolveThemeMode(preference, prefersDark)` returning `light|dark`.
    - `applyThemeMode(mode)` ensuring root class has only one of `light|dark`.
    Call this bootstrap in `web/src/main.tsx` before `ReactDOM.createRoot(...)`.
    Add `web/scripts/check-no-hardcoded-colors.mjs` that recursively scans `web/src/**` and supports wave-aware execution modes:
    - `--mode changed [--base-ref <git-ref>]` (Wave 1/2 gate on touched files only)
    - `--mode baseline-diff --baseline <path>` (Wave 1/2 gate blocks new violations over approved baseline)
    - `--mode strict` (Wave 3 repo-wide gate)
    For `--mode changed`, implement deterministic base-ref fallback strategy:
    1) explicit CLI `--base-ref`,
    2) `CHECK_COLORS_BASE_REF` env var,
    3) `origin/main` when available,
    4) fallback to `git merge-base HEAD HEAD~1`.
    Check must fail with actionable error when none of the options resolve.
    Detection policy must fail on:
    - color literals (`#hex`, `rgb(...)`, `hsl(...)` with numeric channels)
    - forbidden palette utility classes (for example `text-red-600`, `bg-blue-100`, `border-emerald-500/20`)
    - inline style color properties with literal values (`style={{ color: '#fff' }}`, `background`, `borderColor`, and related variants)
    Explicitly allow semantic variable-driven values in both class/inline paths: `var(--token)` and `hsl(var(--token))`.
    Define strict token-definition exception for `web/src/index.css` by region markers only:
    - `/* COLOR_POLICY_ALLOW_START:token-definitions */`
    - `/* COLOR_POLICY_ALLOW_END:token-definitions */`
    Script must still scan whole `index.css` in all modes and only skip literal checks between these markers.
    Define explicit allowlist in the script:
    - non-UI static asset files under `web/src/assets/**`
    Add formal allowlist governance manifest `web/scripts/check-no-hardcoded-colors.allowlist.json` with mandatory fields:
    `path`, `pattern`, `reason`, `owner`, `approved_by`, `expires_at`.
    Entries with missing `expires_at` or expired timestamp must fail the check.
    Produce baseline snapshot file `.planning/phases/06-color-scheme-refresh/06-COLOR-POLICY-BASELINE.json` from initial scan for baseline-diff mode.
    Add fixture tests for policy script robustness:
    - `web/scripts/__fixtures__/check-colors/known-good.tsx` (semantic-only usage, expected PASS),
    - `web/scripts/__fixtures__/check-colors/known-bad.tsx` (literal + forbidden utility + inline literal style, expected FAIL),
    - `web/scripts/check-no-hardcoded-colors.test.mjs` executes checker against both fixtures and asserts exit behavior/messages.
    Script output must print file path + line number for each violation.
    Register script in `web/package.json` as `"check:colors": "node scripts/check-no-hardcoded-colors.mjs"`.
  </action>
