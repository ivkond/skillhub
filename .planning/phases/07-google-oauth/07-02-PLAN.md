# Phase 7 Plan 02: Web Login UX For Google OAuth

<objective>
Интегрировать Google OAuth в web login experience так, чтобы provider rendering оставался data-driven, а поведение redirect было предсказуемо и протестировано.
</objective>

<provider_contract>
Backend contract: `spring.security.oauth2.client.registration.google` публикуется как provider `google`.
UI/API contract: `/api/v1/auth/methods` содержит метод `id=oauth-google`, `methodType=OAUTH_REDIRECT`, `actionUrl=/oauth2/authorization/google?returnTo=%2F`.
Contract checks:
- Backend integration: `AuthControllerTest` (Plan 01/03) фиксирует `google` + `oauth-google` + `actionUrl` consistency.
- Web integration: `login-button.test.ts` и `login.test.tsx` проверяют рендер/redirect именно для `oauth-google` без hardcoded drift.
</provider_contract>

<threat_model>
<asvs_level>1</asvs_level>
<threats>
  <threat id="T-07-02-01" severity="high">UI hardcodes provider routes and diverges from backend action URLs.</threat>
  <threat id="T-07-02-02" severity="medium">Google entry rendering breaks existing providers or fallback behavior.</threat>
  <threat id="T-07-02-03" severity="medium">Localization gaps produce mixed-language login CTA text.</threat>
  <threat id="T-07-02-04" severity="high">Provider id drift between backend `google` and UI `oauth-google` causes broken login button binding.</threat>
</threats>
<mitigations>
  <item>Сохранять единственный source of truth: `provider.actionUrl` from backend.</item>
  <item>Расширить provider-aware tests на multiple providers включая `oauth-google`.</item>
  <item>Обновить i18n keys и проверить оба локаля (en/zh) на login OAuth copy.</item>
  <item>Зафиксировать и протестировать invariant mapping: `google` -> `oauth-google` -> `/oauth2/authorization/google`.</item>
</mitigations>
<block_on>high</block_on>
</threat_model>
