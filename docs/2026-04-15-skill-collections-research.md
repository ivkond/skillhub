# Skill Collections — Research Notes

> Date: 2026-04-15  
> Status: Research / pre-spec  
> Language: RU (рабочие заметки для согласования продукта и реализации)

## 1. Цель

Позволить пользователям **группировать скиллы** по роли или решаемой задаче (например research, backend-java, android), **именовать подборку** и **делиться ею** с другими — без замены существующих механизмов (`skill_tag` как алиасы версий, админские `label_*`, персональные `skill_star`).

## 2. Зафиксированные решения (на текущий момент)

| Тема | Решение |
|------|---------|
| Роли | **Owner** — один на коллекцию; полный доступ (автор / владелец). **Contributor** — много; может **добавлять и исключать скиллы** из подборки; доступ **ограниченный** (точная граница — см. открытые вопросы). |
| Администраторы платформы | По умолчанию **полный доступ** к коллекциям (как к прочим сущностям под governance). |
| Приватные скиллы | **Нельзя добавить** в коллекцию (инвариант; проверка должна совпадать с моделью видимости и доступа в коде — `SkillVisibility`, членство в неймспейсе, `VisibilityChecker` / `SkillQueryService`). |
| Канал управления (MVP) | Коллекции **только через Web UI**; отдельные API-токены / CLI / расширение `RouteSecurityPolicyRegistry` для machine-клиентов **в первую очередь не требуются**. |
| Согласованность | Приоритет — **единая трактовка ролей, видимости и edge cases** до активной разработки; этот файл — база для spec / PLAN. |

## 3. Связь с текущей архитектурой (кратко)

- Новый вертикальный срез: **БД (Flyway)** → **skillhub-domain** (сущности, репозитории, политики) → **skillhub-app** (DTO, AppService, контроллеры) → **skillhub-auth** (маршруты web-сессии) → **web** (роуты, страницы, хуки).
- Поиск (`skillhub-search`), уведомления, публичный API-токены — **опционально позже**, если не входят в MVP.
- Существующие сущности **не переиспользовать как «коллекцию»**: `SkillTag`, `SkillLabel` / `label_definition`, `SkillStar` — другая семантика.

## 4. Идеи к проработке (ещё не решения)

- **Идентификация коллекции в URL:** slug per owner, глобальный числовой id, составной ключ — влияет на уникальные ограничения в БД и SEO/шаринг.
- **Видимость коллекции:** только `public` / `private`, или ещё `unlisted` (по ссылке); влияет на `permitAll` vs `authenticated` в security policy.
- **Порядок скиллов в подборке:** явный `sort_order` vs порядок добавления.
- **Пустая коллекция:** разрешена ли до публикации; отображение для зрителя.
- **Приглашение contributors:** только owner, или делегирование; срок приглашения; отзыв доступа.
- **События жизненного цикла скилла:** skill удалён, заархивирован, скрыт, сменилась видимость — автоматически убрать из коллекции, показать «недоступно», запретить сохранение до исправления.

## 5. Открытые вопросы (нужны ответы до финального spec)

### 5.1 Граница прав Contributor

- Может ли contributor: **переименовывать** коллекцию, менять **описание**, менять **видимость** (public/private), **удалять** коллекцию?
- Может ли contributor: **приглашать / удалять** других contributors?
- Может ли contributor: **менять порядок** скиллов в подборке?
- Нужна ли отдельная роль **viewer** (только чтение состава) для шаринга, или достаточно «публичная ссылка без логина» для public-коллекций?

### 5.2 Модель видимости и аудитория

- Для **public** коллекции: неавторизованный пользователь видит **только публичные** скиллы из состава или отказ в показе всей коллекции, если есть «дыра»?
- Нужен ли режим **«коллекция видна только участникам неймспейса X»** или на первом этапе только user-scoped коллекции?

### 5.3 Инвариант «нельзя добавить приватный скилл»

- Кто считается «субъектом проверки» при добавлении: **тот, кто нажал «добавить»**, или **все** участники должны иметь право видеть скилл?
- Разрешено ли owner добавить скилл, который contributor **не видит** (и наоборот)?

### 5.4 Администраторы

- «Полный доступ» = **только read/write данные**, или ещё **аудит / модерация** (скрытие коллекции, принудительная смена owner)?
- Нужны ли **админские экраны** в MVP или достаточно существующих инструментов governance?

### 5.5 Лимиты и злоупотребления

- Лимиты на число коллекций на пользователя, число скиллов в коллекции, число contributors.
- Нужна ли **модерация** публичных коллекций (название, описание).

### 5.6 Локализация и копирайт

- Язык **названия и описания** коллекции: один язык или i18n-поля с первой версии.

### 5.7 Уведомления

- Уведомлять ли contributor при **изменении состава другим** участником, при **отзыве доступа**, при **удалении коллекции**.
